## 2. 경쟁 제품 분석

### 2.1 Noom

**서비스 개요**

심리학 기반 행동 변화 프로그램과 전담 코치 연결을 결합한 체중 관리 앱. 음식을 칼로리 밀도에 따라 Green / Yellow / Red 3단계로 분류하며, 사용자의 수동 식단 입력과 코치 피드백을 중심으로 운영된다.

**핵심 한계**

| 항목 | 내용 |
|------|------|
| 식단 기록 방식 | 텍스트 검색 기반 수동 입력. 한국 음식 DB 커버리지 부족으로 국내 사용자 정확도 저하 |
| 코칭 개인화 | 코치가 참조하는 데이터는 사용자가 직접 입력한 내용에 한정. 비동기 대화 구조로 실시간 피드백 불가 |
| 질환 특화성 | 서비스 목적이 체중 감량에 집중되어 있으며, 고혈압·당뇨 위험군을 위한 나트륨·혈당 관련 식이 추적 기능 부재 |

### 2.2 Samsung Health

**서비스 개요**

Galaxy Watch 등 삼성 기기와의 연동을 기반으로 걸음 수, 수면, 심박수, 혈압 등 생체 데이터를 수집·시각화하는 종합 헬스 플랫폼. 국내 안드로이드 사용자 기반이 넓고 웨어러블 연동이 핵심 강점이다.

**핵심 한계**

| 항목 | 내용 |
|------|------|
| 식단 기능 완성도 | 식단 기록은 바코드 스캔·검색 입력에 한정되며, Vision AI 기반 자동 분석 기능 미지원 |
| 데이터 활용 | 수집된 생체 데이터(심박수, 수면, 걸음 수 등)가 개별 지표 시각화에 그치며, 식단·운동 데이터 간 통합 인사이트 미제공 |
| 플랫폼 종속성 | Samsung 기기 및 Galaxy Watch 없이는 주요 기능이 제한됨. iOS 및 타 Android 기기 사용자에 대한 동등한 서비스 제공 불가 |

### 2.3 MyFitnessPal

**서비스 개요**

전 세계 2억 명 이상의 사용자를 보유한 식단·운동 통합 관리 앱. 1,400만 개 이상의 식품 DB 기반 칼로리 추적과 바코드 스캔 식품 등록이 핵심 기능이며, 글로벌 식단+운동 관리 앱의 사실상 표준으로 평가받는다.

**핵심 한계**

| 항목 | 내용 |
|------|------|
| 한국 식품 DB 정확도 | DB의 대부분이 서구권 식품 중심으로 구성. 한국 가정식·복합 요리는 사용자 비공식 등록 데이터에 의존하여 영양 수치 신뢰도 저하 |
| 사진 인식 정확도 | 유료 플랜 한정으로 사진 인식 기능 제공되나, 복합 요리의 재료 분리 인식 및 섭취량 추정 정확도가 낮아 사용자의 수동 보정이 필요 |
| 코칭 개인화 | 칼로리 목표 대비 초과 여부 알림 수준에 머물며, 누적 식단 데이터를 기반으로 한 맥락 있는 개인화 피드백 미제공 |

> **인터뷰 근거 (기존 앱 한계 실증)**
> 인터뷰에 참여한 20대 당뇨 환자는 기존 식단 기록 앱을 사용하지 않는 이유로, **기록 행위 자체가 건강 관리 행동의 실질적 변화로 이어지지 않는다**는 점을 지적하였다. 해당 사용자는 단백질·채소류 중심의 식단 원칙을 이미 내재화하고 있었으며, 실제 관리는 식품의 세부 영양 수치 추적이 아닌 **복약 여부 및 운동 수행 여부의 단순 확인** 수준에서 이루어지고 있었다. 이는 기존 앱이 제공하는 상세 기록 중심의 인터페이스가 실제 만성질환 관리 행동 패턴과 괴리가 있음을 시사한다.

## 3. On-Care 차별성 분석

### 3.1 Vision AI 2단계 식단 인식 파이프라인 🔍

기존 앱들의 식단 기록 방식은 수동 입력(Noom, Samsung Health) 또는 사진 인식 기능을 제공하더라도 정확도가 낮고 수동 보정이 요구되는 구조(MyFitnessPal)로 수렴한다. 이는 식단 관리 지속성을 저해하는 핵심 요인이다.

인터뷰 결과, 만성질환 환자가 기록 앱을 이탈하는 주요 원인은 기록에 투입되는 노력 대비 실질적 피드백의 부재에 있었다. 기록 행위가 행동 변화로 연결되지 않는 구조에서는 사용자가 앱 사용을 지속할 유인을 상실하게 된다. On-Care는 이 문제를 해결하기 위해 **YOLOv8 + Gemini Vision API 2단계 파이프라인**을 설계하였다.

- **1단계 (YOLOv8):** 경량 모델로 음식 이미지 여부를 빠르게 판별하여 비음식 이미지에 대한 불필요한 API 호출을 차단. 응답 속도 및 운영 비용 효율화에 기여한다.
- **2단계 (Gemini Vision API):** 음식으로 확인된 이미지에 한하여 음식 종류 식별 및 추정 섭취량 분석을 수행한다.
- **DB 매핑 (공공데이터 식품영양성분 DB):** 분석 결과를 한국 식품 기준 공공 데이터에 매핑하여 칼로리·탄수화물·단백질·지방·나트륨 수치의 정확도를 확보한다.

MyFitnessPal이 서구권 중심 DB에 의존하는 한계를 공공데이터 활용으로 극복하고, 사진 한 장으로 기록이 완결되는 사용자 경험을 목표로 한다.

### 3.2 RAG 기반 개인화 AI 코칭 🤖

Noom의 코칭은 사용자 입력 데이터에 종속되며, Samsung Health와 MyFitnessPal은 개인 건강 이력을 참조하는 AI 코칭 기능 자체가 부재하다.

인터뷰를 통해 만성질환 환자의 식단 관리 구조를 구체적으로 파악할 수 있었다. 해당 사용자는 허용 식품군의 범주(단백질·채소류 등)는 이미 인지하고 있었으나, 범주 내 구체적인 식품 선택은 별도의 판단 기준 없이 이루어지고 있었다. 동일 식품을 장기 반복 섭취하는 경우 대체 식품에 대한 정보가 필요하나 이를 제공하는 수단이 부재하였으며, 질환 특성상 혈당 급등을 유발하거나 나트륨 부담이 높은 식품군에 대한 **맥락적 경고 및 대안 안내** 또한 충족되지 않고 있었다.

이러한 니즈는 단순 칼로리 알림이나 일반적 영양 정보 제공으로는 충족될 수 없다. 사용자의 누적 식단 이력과 질환 정보를 동시에 참조해야만 실질적인 조언이 가능하기 때문이다.

On-Care는 **LangChain + Pinecone + GPT-4o 기반 RAG(Retrieval-Augmented Generation) 파이프라인**을 통해 이 한계를 해결한다.

- 사용자의 인바디 이력, 식단 기록, 운동 로그, 질환 정보, 목표 설정 데이터를 Pinecone에 벡터로 저장한다.
- 사용자 질의 입력 시, 의미적으로 연관된 개인 데이터를 검색하여 GPT-4o의 프롬프트 컨텍스트로 주입한다.
- 결과적으로 일반적 정보 제공이 아닌, **사용자 개인의 누적 기록을 근거로 한 맞춤 식단·운동 조언**을 생성한다.
- 만성질환 위험군에 대해서는 **질환 정보 기반 섭취 제한 식품 경고** 및 **반복 식단 패턴 감지 후 동일 식품군 내 대체 식품 추천** 기능을 RAG 컨텍스트에 포함하여 질환 특화 코칭을 구현한다.

설계 과정에서 팀이 집중한 핵심 과제는 벡터 DB에 입력할 데이터의 범위와 임베딩 전략 결정이었다. 데이터 범위가 지나치게 넓으면 검색 노이즈가 증가하고, 좁으면 개인화 효과가 감소한다. 이에 따라 질문 유형별 연관 데이터 우선순위 정의 및 검색 파라미터 튜닝을 핵심 구현 과제로 설정하였다.

### 3.3 오프라인 운동 환경 연결 — 헬스장·트레이너 연동 🏋️

경쟁 제품 3개 모두 오프라인 운동 인프라(헬스장, 트레이너)와의 연결 기능을 제공하지 않는다. 사용자가 헬스장 탐색, 예약, 트레이너 상담을 위해 별도 앱이나 채널로 이탈해야 하는 구조는 서비스 연속성을 저해한다.

On-Care는 **카카오맵 API 기반 위치 기반 헬스장 검색·예약** 및 **트레이너 인앱 채팅** 기능을 통합한다. 특히 트레이너 연결 시 사용자의 누적 건강 데이터 요약본을 자동으로 전달하는 기능을 포함하여, 트레이너의 초기 상담 효율성을 높이는 동시에 앱 내 데이터가 오프라인 운동 지도에까지 연결되는 경험을 제공한다.

### 3.4 고혈압·당뇨 위험군 특화 설계 🎯

경쟁 제품 3개 모두 체중 감량 또는 일반 건강 관리를 주요 목적으로 설계되어 있으며, 만성질환 위험군에 특화된 기능을 제공하지 않는다.

인터뷰를 통해 확인된 바에 따르면, 외래 치료 수준의 당뇨 환자는 특정 식품만을 섭취해야 하는 제약보다는 **섭취를 제한해야 하는 식품군이 존재하는 형태**로 식단을 관리한다. 허용된 식품군 내에서의 선택 다양성은 충분히 확보되어 있으며, 관리의 핵심은 혈당 급등을 유발하는 식품 회피와 나트륨·지방 섭취 조절에 있다. 기존 앱들은 이러한 만성질환 관리의 구조적 특성을 반영하지 못한 채 일반적 칼로리 추적에 머물고 있다.

On-Care는 서비스 기획 단계부터 타겟을 **2030 고혈압·당뇨 위험군**으로 명확히 설정하였다. 이에 따라 단순 칼로리 추적을 넘어 나트륨 섭취량 추적 및 경고, 혈당 지수(GI) 기반 식품 분류, 야식·불규칙 식사 패턴 감지 등 만성질환 위험 관리에 직결되는 기능들을 핵심 설계 요소로 포함하였다. 체중 감량을 목적으로 설계된 앱에서 부수 기능으로 다뤄질 수밖에 없는 항목들을 On-Care에서는 서비스의 중심에 위치시킨다.

## 4. 결론

On-Care는 기존 헬스케어 앱들이 공통적으로 해결하지 못한 네 가지 문제  
**① 식단 기록의 높은 진입 장벽**  
**② 개인 건강 이력을 참조하지 않는 코칭**  
**③ 오프라인 운동 환경과의 단절**  
**④ 만성질환 위험군 특화 기능 부재**  
를 동시에 해결하는 것을 목표로 설계되었다.

이 문제들은 문헌 분석에 그치지 않고 실제 20대 당뇨 환자 인터뷰를 통해 실증적으로 확인되었다. 인터뷰 결과는 기존 앱이 기록 도구에 머물러 실질적 행동 변화를 유도하지 못하고 있음을 보여주었으며, 만성질환 관리에 필요한 것이 정밀한 영양 수치 추적이 아닌 **누적 패턴 기반의 맥락적 조언**임을 확인하였다.

각 기술적 선택(YOLOv8 필터링, RAG 아키텍처, 카카오맵 연동, 공공데이터 DB 매핑)은 단순한 기능 추가가 아닌, 위의 문제 해결을 위한 팀의 설계 판단에 근거한다. 이를 통해 On-Care는 **기록 도구에 머무르는 기존 앱과 달리, 사용자의 실질적 건강 행동 변화를 유도하는 통합 헬스케어 플랫폼**을 지향한다.
